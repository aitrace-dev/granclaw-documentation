API keys that never touch disk.

Every agent you run needs credentials of some kind — an API key to call a service, a bot token to send Telegram messages, a password to authenticate with an internal tool. GranClaw gives you a secure place to store these and a deliberate guarantee about how they are handled: they are never written to any file, never committed to version control, and never logged anywhere.

## The Secrets panel

You add secrets in your agent's settings. Open the dashboard, navigate to your agent, and find the Secrets tab in Settings. Each secret has a name and a value. The name is what the agent and your workflows use to refer to the credential — something like `OPENAI_API_KEY` or `TELEGRAM_BOT_TOKEN`. The value is the actual secret itself.

You can add as many secrets as an agent needs. Each one is stored encrypted, isolated to that agent, and never displayed in plain text after you save it.

## How secrets stay safe

When GranClaw starts an agent process, it reads the agent's secrets from the encrypted store and injects them as environment variables directly into that process at spawn time. From that point on, the agent can use them the way any program uses environment variables — by name, without knowing the underlying value.

The secret never leaves the encrypted store in any other way. It is not written to the agent's workspace directory. It is not written to any log. It is not included in conversation history. It is not committed if you use version control for your workspace.

<Callout type="tip">
Unlike `.env` files, your secrets are never on disk in plain text. A `.env` file is a file — it can be accidentally committed, copied, or read by another process. GranClaw secrets live in an encrypted store and exist as environment variables only for the duration of the agent process.
</Callout>

## Scoping

Secrets are per-agent. An agent can only read its own secrets — there is no shared secrets store across agents. If you have two agents and both need an API key for the same service, you add the secret to each of them separately.

This isolation is intentional. It means a compromised or misbehaving agent cannot access credentials belonging to a different agent running on the same machine. Each agent's secret environment is its own.

## Rotating a key

To rotate a secret, open the agent's Secrets tab, find the entry, and update the value. The change takes effect the next time the agent process starts — stop and restart the agent to pick up the new credential.
